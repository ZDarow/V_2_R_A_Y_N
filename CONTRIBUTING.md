# Участие в проекте

Спасибо за интерес к проекту! Вот правила и рекомендации для контрибьюторов.

## Code of Conduct

Проект придерживается принципов открытости и взаимного уважения.
Оскорбления, троллинг и неуважительное общение недопустимы.

## Как помочь

### Сообщить об ошибке
- Откройте Issue с меткой `bug`
- Опишите: что ожидали, что произошло, окружение (ОС, версия v2rayN/Xray)
- Приложите лог диагностики: `scripts/netcheck.sh` или `scripts/detect-block-type.sh`

### Предложить улучшение
- Откройте Issue с меткой `enhancement`
- Опишите проблему и предлагаемое решение

### Отправить PR
1. Форкните репозиторий
2. Создайте ветку: `feature/что-делаете` или `fix/что-чините`
3. Внесите изменения
4. Убедитесь, что CI проходит (shellcheck, bash -n, BATS тесты)
5. Откройте Pull Request на ветку `main`

## Требования к коду

### Shell-скрипты
- `set -uo pipefail` (кроме скриптов, где нужен graceful fail)
- Bash syntax: `bash -n` без ошибок
- Shellcheck: `shellcheck -S warning -e SC1091` без ошибок
- Цвета: `R`, `G`, `Y`, `B`, `C`, `N` (см. существующие скрипты)
- Функции: `ok()`, `fail()`, `warn()`, `info()`, `step()`
- Используйте `lib/common.sh` для общего функционала

### JSON-конфиги
- Валидный JSON: `python3 -m json.tool`
- Следуйте схеме: `spec/json-schema/`

### Тесты (BATS)
- Добавляйте тесты для новой функциональности в `spec/`
- Запуск: `bats spec/*.bash`

### Документация
- Изменения API/CLI отражайте в README.md
- Крупные изменения — в docs/

## Названия веток
- `feature/<what>` — новая функциональность
- `fix/<what>` — исправление
- `docs/<what>` — документация
- `chore/<what>` — обслуживание

## Коммиты
- Язык: русский
- Формат: `<тип>: <краткое описание>`
- Типы: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `style`, `perf`, `ci`
- Длина заголовка ≤ 72 символа
- Тело коммита с новой строки, поясняет «почему»
